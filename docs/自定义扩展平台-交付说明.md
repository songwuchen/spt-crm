# SPT-CRM 自定义扩展平台 — 交付说明

> 配套文档：[自定义扩展平台-开发计划](自定义扩展平台-开发计划.md)（现状评估与总体设计）。
> 本文面向交付/演示与运维，说明**已交付的四大能力**、**使用入口与操作步骤**、**旧审批引擎灰度替换方案**，以及**尚未覆盖的能力与后续规划**。

---

## 一、总览：四大能力已交付

| 能力 | 说明 | 菜单入口 |
|---|---|---|
| **① 自定义字段** | 各业务实体（客户/商机/线索/联系人/订单/合同/报价/工单/回款）的扩展字段，统一由**表单引擎**驱动设计与渲染，值仍存业务表 `custom_fields_json` 列。 | 扩展平台 → 字段管理 |
| **② 自定义表单** | schema 驱动的表单引擎：模板/版本/实例，29 种字段类型，公式联动、必填校验、流水号、明细子表。 | 扩展平台 → 表单中心 |
| **③ 可视化审批流程** | `@xyflow` DAG 流程设计器 + 运行时引擎（或签/会签/顺序、条件分支路由），可绑定表单或**既有业务单据**，与旧审批引擎并存、按业务类型灰度切换。 | 扩展平台 → 流程管理 / 审批中心 |
| **④ 可配置仪表盘** | `react-grid-layout` 拖拽网格 + `echarts`，对表单数据与 CRM 业务数据（客户/线索/订单/商机/合同/工单）运行时聚合。 | 扩展平台 → 仪表盘 |

**架构基线**：JSONB schema 驱动，**无动态建表、无 EAV**。
`lc_form_template(+version)` 存 `field_definitions/layout_definition/rule_definitions` 三个 JSONB；
`lc_form_instance` 存 `form_data` JSONB 并在提交时**快照 field_definitions**，保证历史记录始终可渲染。

## 二、使用指引

### ① 自定义字段（统一到表单引擎）
1. 进入 **扩展平台 → 字段管理**，选择业务实体（如「客户」）。
2. 用表单设计器增删字段（拖拽、设类型、必填、选项…），**发布**。
3. 该实体的新建/编辑/详情页会自动出现「扩展字段」区块（同一套 FormRenderer 渲染）。
4. 商机列表会把已发布的扩展字段渲染成列。

> 说明：旧 `custom_field_defs` 自定义字段系统已下线（详见第四节）。历史扩展字段值仍在各业务表 `custom_fields_json` 中；如需在新引擎里继续展示，在「字段管理」中按同名字段重建即可。

### ② 自定义表单
1. **扩展平台 → 表单中心 → 新建表单**（或**从模板库新建**，见下）。
2. 在设计器中拖拽字段、配置公式/联动/流水号/明细子表，**发布**。
3. 「填报」录入数据，「数据」查看列表，支持**导出 Excel**。

**模板市场（一键安装）**：表单中心 → **从模板库新建**，内置 5 个开箱即用表单：
请假申请 / 报销申请 / 用章申请 / 物品领用 / 采购申请。安装后为草稿，可继续增删字段、绑定审批流再发布。

### ③ 可视化审批流程
1. **扩展平台 → 流程管理 → 新建流程**，二选一绑定：
   - **绑定表单**：该表单提交即自动发起此审批流；或
   - **绑定业务类型**：订单/线索/工单/报价单/合同/变更单/方案 —— 该业务单据提交审批时走此可视化流程。
2. 在画布（`@xyflow`）中编排开始 → 审批（或签/会签/顺序）→ 条件分支 → 抄送 → 结束，配置审批人，**发布**。
3. **审批中心**处理待办；流程完成/驳回后**自动回写业务单据状态**。

### ④ 可配置仪表盘
1. **扩展平台 → 仪表盘 → 新建**，拖拽添加图表卡片。
2. 数据源可选**自定义表单**或**CRM 业务数据**（客户/线索/订单/商机/合同/工单）。
3. 选维度（含日期年/月粒度）与指标（计数/求和/均值…），保存后网格布局自适应。

## 三、旧审批引擎 → 可视化引擎：按 biz_type 灰度切换

**机制**：各业务提交审批时优先调用新引擎 `start_for_biz(biz_type, biz_id)`：
- 若该 `biz_type` 已**绑定并发布**可视化流程 → 走新引擎，承载 `(biz_type, biz_id)`，完成/驳回后由 `wf_biz_writeback` 把结果回写业务表状态列；
- 否则返回 `None` → 回退旧 `auto_trigger_approval`（旧策略引擎）。

> 「是否存在已发布绑定」即灰度开关，逐个业务类型平滑切换，**可随时回滚**（删除/停用绑定即回到旧引擎）。

**已接入的 6 个提交点 / 7 类业务**：`order`、`service_ticket`、`quote_version`、`contract_version`、`change_request`、`solution`（+ `lead` 走既有线索审核流）。
回写值与旧引擎 `_on_approval_completed/_rejected` 语义一致（`approved/rejected`；订单为 `confirmed/cancelled`）。

## 四、Phase 5 打磨内容（本轮）

1. **旧自定义字段系统下线**：商机列表迁到新引擎读取；移除 设置页「自定义字段」Tab、`CustomFieldsPanel` 组件、`/api/v1/custom-fields` 接口。`CustomFieldDef` 模型与 `custom_field_defs` 表**保留（不做破坏性迁移）**，历史定义数据仍在库中。
2. **审批灰度切换**：见第三节（6 提交点接入 + 回写覆盖全部 biz_type）。
3. **仪表盘数据源扩展**：新增 商机/合同/工单；软删过滤按实体标记（合同/工单表无 `is_deleted`）。
4. **表单数据导出**：`GET /api/v1/lc/form-instances/export`，按当前筛选导出 Excel（列＝业务编号/标题/状态/创建时间＋各字段），遵循数据范围。
5. **模板市场**：内置模板一键安装。
6. **性能**：表单实例列表/导出的主查询已由复合索引 `ix_lc_form_instance_tpl_created(tenant_id, template_id, created_at)` 覆盖，无需额外加索引。

## 五、字段级权限（已交付）

按角色控制表单字段的「可见 / 只读 / 可编辑」。规则挂在字段上：`visible_roles`（谁可见）、`edit_roles`（谁可编辑，值为角色 code，空=不限制）；可见但不可编辑 → 只读。

- **配置**：表单设计器每个字段的「权限」弹窗里选「可见角色 / 可编辑角色」。
- **后端（权威边界，自定义表单完整强制）**：读取剔除隐藏字段的定义与值、不可编辑字段标记只读；写入丢弃用户对不可编辑/隐藏字段的改动（保留原值）；导出列同样按可见性过滤。设计器预览与审批查看不受限（看全量）。
- **前端（UX）**：`FormRenderer` 依据登录者角色隐藏/只读渲染；必填校验跳过对本人隐藏的字段。
- **当前边界**：
  - **自定义表单（form_instance）**：读写**双向**后端强制（读取剔除隐藏字段值、写入丢弃不可编辑字段）。
  - **实体自定义字段**（客户/商机/线索/订单/工单/联系人的 `custom_fields_json`）：**写入**已后端强制（6 类实体的新建/编辑丢弃用户对不可编辑/隐藏字段的写入）；**读取隐藏**已覆盖各实体的**列表 + 详情 + 联系人子列表**主读取路径（按查看者角色剔除隐藏字段值，每请求只查一次字段定义，无配置零开销）。**残留**：创建/更新等写操作的**响应体**暂未裁剪（操作者查看自己刚写入的记录，更新响应可能回显不可编辑字段的原值）——如需彻底闭合可后续补齐。

## 六、尚未覆盖 / 后续规划

- **移动端（/m 分区）表单填报与审批**：当前扩展平台页面为桌面 antd 组件。移动端需响应式改造或引入 antd-mobile，并打通移动审批中心。（审批待办的钉钉/企微推送已具备，见集成框架。）
- **实体自定义字段读取裁剪收尾**：列表/详情/联系人子列表已后端裁剪；仅剩创建/更新等写操作的响应体未裁剪（残留，见第五节）。
- **审批引擎增强**：剩 并行网关 / 会签加签 / 超时自动。已落地：**代理审批**（审批中心「我的代理」设置某时段由他人代审，代理人在其待办看到并可代为通过/驳回，留痕标注）、**退回指定节点**（审批人把流程退回到某审批节点重审，作废当前待办后重新激活目标节点，流程保持进行中）。

## 八、AI 模型接入与向量知识库（本次交付）

把 AI 从"内置模拟（mock）"升级为**后台可配置的真实大模型接入**，并为知识库接入**pgvector 语义检索**。

### 8.1 后台配置（系统设置 → AI模型）
- **对话/分析模型**：供应商可选 **通义千问(DashScope) / DeepSeek / OpenAI / Claude / 自定义(OpenAI 兼容)**；配置 Base URL、模型名、API Key。切换供应商自动带出预设 Base URL 与常用模型。
- **向量嵌入模型**：供应商可选 **通义 text-embedding / OpenAI Embeddings / 自定义**；统一输出 **1024 维**（与向量库列对齐）。
- **启用开关**：关闭时分析回退内置模拟、知识库回退关键词检索。
- **密钥**：AES 加密存储（`ENCRYPTION_KEY`），响应掩码为 `***`，回填空值不覆盖已存密钥。
- **测试连接**：分别对对话/嵌入模型做真实联通性 ping。
- 权限 `role:manage`；接口 `GET/PUT /api/admin/v1/tenant/ai-settings`、`POST .../test-chat`、`POST .../test-embedding`。

### 8.2 运行时接线
- `ai_engine._call_llm` 优先级：**租户后台配置 > 环境变量 `AI_PROVIDER` > mock**。通义/DeepSeek/OpenAI 走 OpenAI 兼容 `/chat/completions`，Claude 走原生 Messages API。
- AI 中心 `POST /api/v1/ai/analyze`（风险/画像/赢率/行动/报价/合同审核）与相似商机分析均按租户配置调用真实模型；token 用量与成本按调用回填 `AiTask`，并计入月度 AI 预算配额（`TenantAiBudget`，超额拦截）。

### 8.3 向量知识库（pgvector）
- **数据库**：生产 Postgres 由 `postgres:16-alpine` 切换到 **`pgvector/pgvector:pg16`**（同 PG16，数据卷复用）。迁移 `ai01c2d3e4f5` **环境自适应**：仅当数据库支持 pgvector（`pg_available_extensions` 含 `vector`）时创建 `knowledge_chunks.embedding vector(1024)` 列 + **HNSW 余弦索引**；否则只建设置表（CI/普通 postgres 不受影响）。
- **写入**：知识库文档创建/更新时，对每个 chunk 调嵌入模型生成向量并写入（尽力而为，失败不阻塞文档保存）。
- **检索**：`search_chunks` 优先 `ORDER BY embedding <=> query` 余弦近邻；未启用嵌入或无向量列时回退关键词 `ilike` 匹配。
- **验证**：通义 `text-embedding-v3` 实测——相关句对余弦 ≈0.91、无关句对 ≈0.35，语义区分正确。

### 8.4 上线注意（生产 DB 镜像切换）
生产内网无 Docker Hub，pgvector 镜像需经 Harbor 或 `docker save|ssh|load` 落到生产，替换 db 服务镜像并重启（**重启 db 有短暂停机**）。若本迁移曾在无 pgvector 环境执行过（未建向量列），切镜像后需幂等补跑同样 DDL（`CREATE EXTENSION vector` + `ADD COLUMN embedding` + 建 HNSW 索引），进程重启后自动识别到向量列。
