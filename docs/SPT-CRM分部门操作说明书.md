# SPT-CRM 分部门操作说明书

> 本手册站在**各业务部门**的角度，说明每个部门在系统里**做什么、点哪里、上下游怎么衔接、系统会不会自动产生待办/通知**。
> 与 `docs/振动筛制造-CRM操作手册.md`（全流程线性演示）互补：那本讲"一条单子从头走到尾"，本册讲"每个部门各自怎么干、彼此怎么交接"。
>
> 适用版本：截至 2026-06。所有结论均对照实际代码，章节末附"已知限制"。

---

## 一、整体角色 / 部门蓝图

权限模型三层：**权限(Permission，全局)** → **角色(Role，按租户，带数据范围 data_scope)** → **用户(User，可多角色、多部门)**。
系统内置 16 个职能角色（`backend/scripts/seed_function_roles.py`，需 `python -m scripts.seed_function_roles --apply` 写入），对应下列部门。

| 部门 / 团队 | 角色 code | 数据范围 | 可写模块 | 只读 |
|---|---|---|---|---|
| **销售部** | sales_rep 销售专员 | 本人 | 客户/联系人/线索/商机/报价 | 方案、合同、交付、回款、订单、产品、标书 |
| | sales_manager 销售主管 | 本部门(含子部门) | 同上 + 提成查看 + **审批** | |
| | sales_director 销售总监 | 全租户 | 同上（全公司） | |
| **商务 / 标书** | biz_support 商务标书专员 | 本部门 | 标书(全)、协助报价 | 客户、商机、合同、产品 |
| **售前 / 技术 / 研究院** | design_engineer 方案设计工程师 | 本人 | 方案、技术变更 | 客户、商机、报价、合同、产品 |
| | tech_chief 技术总工/评审 | 全租户 | 方案、变更 + **审批** | 全局技术只读 |
| **生产 / 交付** | production 生产/交付专员 | 本部门 | 交付里程碑、订单 | 客户、商机、合同、产品 |
| | production_manager 生产主管 | 本部门 | 交付/订单(全) + **审批** | + 报价 |
| **采购** | procurement 采购专员 | 本部门 | 订单、产品 | 商机、合同、交付 |
| **财务** | finance 财务专员 | 全租户 | 回款、清欠、提成、保函 | 客户、商机、报价、合同、订单 |
| | finance_manager 财务主管 | 全租户 | + 合同财务条款 + 提成政策 + **审批(含管理)** | |
| | collection_officer 清欠专员 | 全租户 | 应收清欠、回款登记 | 客户、合同 |
| **法务** | legal 合同法务 | 全租户 | 合同(编辑+**签署**) + **审批** | 客户、商机、报价 |
| **客户服务部 (售后)** | service_engineer 售后工程师 | 本人 | 工单、实测数据、设备档案 | 客户、合同、产品 |
| | service_manager 售后主管 | 本部门 | 工单(全) + **审批** | |
| **管理层** | executive 高管(只读) | 全租户 | 无（全模块只读）+ **审批** + 审计 | 全部 |
| **系统管理员** | admin 系统管理员 | 全租户(绕过) | **全部**：用户/角色/部门/租户/系统配置/审计 | 全部 |

**数据范围 data_scope（`backend/app/common/data_scope.py`）**：`self` 只看自己负责/创建/被共享/被加为商机成员的数据；`dept` 看本部门子树成员的数据；`all` 看全租户。`admin` 角色绕过限制。
**商机团队成员（ProjectMember）**：一个商机可加入多名成员，带标签 `售前/商务/交付/财务/项目经理` 和读写权限(view/edit)。该标签仅作团队分工标识，真正读写由成员的 `permission` 字段决定；**被加为成员即可看到该商机**（即使超出本人数据范围）。

## 二、跨部门主流程衔接（一张图看懂交接）

```
[销售部] 录线索 Lead ──qualify──▶ 生成客户 Customer
                                      │（销售手动建商机并关联客户）
                                      ▼
[销售部] 商机 Opportunity（阶段闸门，逐级解锁）
   S1 线索确认
   S2 需求分析     闸门：已关联客户
   S3 方案报价     闸门：已填写关键需求          ◀── [售前] 填关键需求
   S4 商务谈判     闸门：有方案 + 方案已审批 + 有报价
        ├── [售前] 出方案 Solution（评审/批准）
        └── [销售/商务] 出报价 Quote ──提交──▶ [审批] 报价审批
   S5 合同签署     闸门：有报价 + 已传附件
        └── [商务] 由报价生成合同 Contract ──提交──▶ [法务/财务] 合同审批 ──▶ [法务] 签署
   S6 交付验收     闸门：有合同 + 金额>0
        │
   赢单(won)：必须存在≥1 份"已签署"合同
        │
        ├──▶ [生产/交付] 交付里程碑 Delivery + ERP 订单
        ├──▶ [财务] 合同付款条款 ──生成──▶ 回款计划 ──对账──▶ 回款记录
        └──▶ [售后] 工单 / 变更 / 续约
```

**关键交接点（谁交给谁）**

| 交接 | 上游部门(动作) | 下游部门(承接) | 触发的衔接物 |
|---|---|---|---|
| 线索→客户 | 销售(转化线索) | 销售 | 生成 Customer（线索 owner 带过去） |
| 需求确认 | 销售/售前(填关键需求) | — | 解锁 S3 闸门 |
| 方案 | 售前(出方案、改状态为已批准) | 销售 | 解锁 S4 闸门(方案已审批) |
| 报价 | 销售/商务(提交报价) | 审批人 | 自动发起报价审批 |
| 合同 | 商务(由报价生成合同、提交) | 法务/财务(审批)→法务(签署) | 自动发起合同审批；签署解锁赢单 |
| 回款 | 财务(在合同详情"生成回款计划") | 财务(登记回款→自动对账) | 回款计划/记录 |
| 交付 | 生产(建里程碑) | 生产 | 交付进度 |
| 售后 | 销售/客服(建工单) | 售后工程师(被指派) | 工单分配通知 |

## 三、待办与通知机制（核心：系统会不会产生待办、发给谁）

> ⚠ 系统**没有统一待办中心**。一个人的"待办"分散在 4 处，必须分别查看：

| 入口 | 内容 | 数据来源 |
|---|---|---|
| 🔔 右上角铃铛 / 通知中心 | 各类站内信通知（实时 WebSocket 推送） | `notifications` 表 |
| ✅ **审批中心 →「待我审批」** | 真正需要操作的审批待办 | `approval_tasks` 表 |
| 📋 待办任务页 / 移动端「待办」 | 手工任务清单（现在分配会通知，见修复） | `user_tasks` 表 |
| 📊 工作台 / 仪表盘卡片 | 我负责的"停滞商机/待跟进线索/待审批"实时聚合（不落库、不发通知） | 实时聚合查询 |

### 3.1 待办/通知生成对照表（发生事件 → 是否产生待办 → 发给谁）

**A. 业务操作即时触发（站内通知）**

| 业务事件 | 是否产生待办/通知 | 接收人 | 通知类型 |
|---|---|---|---|
| 商机阶段推进 | ✅ 通知 | 商机负责人（非操作人时） | 阶段推进 |
| 商机赢单 / 丢单 | ✅ 通知 | 商机负责人 | 系统 |
| 合同签署 | ✅ 通知 | 合同创建人 | 合同签署 |
| 工单分配 / 改派 | ✅ 通知 | 新受理人 | 工单分配 |
| 录入回款记录 | ✅ 通知 | 商机负责人 | 收到回款 |
| 活动记录中 @ 某人 | ✅ 通知 | 被 @ 的人 | 提及我 |
| **分配任务给他人**（本次修复） | ✅ 通知 | 被分配人 | 任务分配 |
| **线索分配/批量改派给他人**（本次修复） | ✅ 通知 | 新负责人 | 线索分配 |
| **客户转交给他人**（本次修复） | ✅ 通知 | 新负责人 | 客户分配 |

**B. 审批流（真正的"待办"，进审批中心）**

| 业务事件 | 待办 | 接收人 | 说明 |
|---|---|---|---|
| 提交审批（顺序） | ✅ 审批任务 + 通知 | **第 1 个审批人** | 逐级激活，前一个通过才轮到下一个 |
| 提交审批（并行/任一） | ✅ 审批任务 + 通知 | **全部审批人** | 并行=全通过；任一=有人通过即过 |
| 报价/合同/变更 提交 | ✅ 自动发起审批 | 按审批策略解析的审批人 | 需配置 `ApprovalPolicy`，否则不发起（见限制） |
| 审批通过 / 驳回 | ✅ 通知 | 发起人 | 审批已决定 |
| 撤回 / 转交审批 | ✅ 通知 | 相关审批人 | — |

**C. 定时提醒（reminder_worker，每 5 分钟扫描）**

| 触发条件 | 接收人 | 通知类型 |
|---|---|---|
| 商机 7 天无活动（S1–S4） | 商机负责人 | 系统(跟进停滞) |
| 回款计划 7 天内到期 | 商机负责人 | 回款提醒 |
| **回款计划已逾期**（本次修复，接入自动） | 商机负责人 | 回款逾期 |
| **应收账款逾期 90 天以上**（本次修复，接入自动） | 客户负责人 | 应收逾期 |
| **保函即将到期**（本次修复，接入自动） | 保函负责人 | 保函到期 |
| 审批超过 SLA 时限 | 当前审批人 | 审批超时 + 升级(催办/自动通过) |
| 合同 30 天内到期 | 商机负责人 | 合同到期 |
| 活动下次跟进日 1 天内 | 活动创建人 | 跟进提醒 |
| 客户长期无活动（按客户等级配置天数） | 原负责人 | 系统 + **自动释放到公海** |
| 定时报表（按租户配置时刻） | 配置的接收人 | 定时报表 |

> ⚠ **部署前提**：上述 C 类全部依赖 `reminder_worker` 独立进程运行（`python -m app.workers.reminder_worker`，docker-compose 里是 `reminder` 服务）。**该进程若没起，所有定时提醒全部失效，且无降级、无告警。**

### 3.2 闸门拦截（不是待办，但会挡住流程）

商机推进到下一阶段时若不满足闸门条件，会被拦截并提示缺失项（不产生待办，需操作人自行补齐）：

| 进入阶段 | 必须满足 |
|---|---|
| S2 需求分析 | 已关联客户 |
| S3 方案报价 | 已填写关键需求（商机编辑页「关键需求」） |
| S4 商务谈判 | 有方案 + **方案状态=已批准** + 有报价 |
| S5 合同签署 | 有报价 + 已上传项目附件 |
| S6 交付验收 | 有合同 + 预期金额>0 |
| 标记赢单 | 至少 1 份**已签署**合同 |

此外：商机关联的报价/合同存在**待审批或被驳回**的审批时，阶段推进会被锁定，直到审批完成或撤销（可由管理员强制推进）。

## 四、各部门操作指引

### 1. 销售部（sales_rep / sales_manager / sales_director）
**菜单**：客户管理、客户公海、联系人、线索管理、商机管理、报价管理、跟进记录、销售目标、报表中心。
**日常**
1. **线索**：`线索管理 → 新建`。系统按公司/联系人/行业/预算自动算线索评分。
2. **转化**：线索详情 →「转化」→ 自动生成客户（线索负责人带过去）。⚠ 转化**只生成客户，不会自动建商机**，需手动 `商机管理 → 新建` 并选客户。
3. **商机推进**：商机详情顶部「推进到 Sx」。每步受闸门校验（见 3.2）。S3 前务必在商机编辑页填「关键需求」。
4. **报价**：商机详情「报价」页或报价管理新建；定稿后将版本状态改为"提交"→ 自动进入审批。
**上游/下游**：上游=自己录入/线索池；下游=售前(方案)、商务(合同)、财务(回款)。
**会收到的待办**：阶段推进、赢单/丢单、收到回款通知；被分配线索/客户/任务的通知；停滞/到期/跟进等定时提醒；作为审批人时进审批中心。
**主管额外**：`dept`/`all` 数据范围可看下属数据；是报价/合同等审批人；可在「销售目标」设定团队目标（需 `dashboard:view`，本次修复后已授予销售主管/总监/高管）。

### 2. 商务 / 标书（biz_support）
**菜单**：标书管理（全权）、报价管理（协助编辑）、客户/商机/合同/产品（只读）。
**日常**：维护标书；协助销售完善报价；**由报价生成合同**（合同管理或商机详情「合同」页）。
**下游**：合同 → 法务/财务审批 → 法务签署。

### 3. 售前 / 技术 / 研究院（design_engineer / tech_chief）
**菜单**：方案管理、变更管理；客户/商机/报价/合同/产品只读。
**日常**
1. 在被指派的商机下 `方案` 页新建方案，编辑配置选型清单 + 风险清单，迭代版本。
2. 评审通过后把方案**状态改为"已批准"**——这是 S4 闸门「方案已审批」的前提。
3. 技术总工(tech_chief)可作为方案/变更的审批人。
**注意**：售前靠"被加为商机成员/指派"看到对应商机（数据范围=self）。

### 4. 生产 / 交付（production / production_manager）
**菜单**：交付里程碑、订单管理；客户/商机/合同/产品只读。
**日常**：商机进入交付阶段后，在商机详情「交付」页建里程碑（计划/实际日期、状态、负责人），维护 ERP 订单映射。生产主管可审批交付类事项。
**上游**：合同签署 / 赢单后启动交付。

### 5. 采购（procurement）
**菜单**：订单管理、产品目录；商机/合同/交付只读。
**日常**：按交付需求维护采购订单、产品信息。

### 6. 财务（finance / finance_manager / collection_officer）
**菜单**：回款管理、应收清欠、业务提成、保函管理；合同/报价/订单只读（财务主管可编辑合同财务条款）。
**日常**
1. **生成回款计划**：合同详情「付款条款」区 →「生成回款计划」。系统按条款预填（显式金额优先，否则按比例×合同金额反算），可补到期日、挂交付里程碑（进度款），支持"覆盖本合同上次生成 / 追加"。
2. **登记回款**：商机详情「回款」页录入到账记录；若关联了回款计划，对应计划**自动标记为已回款**。
3. **清欠**：应收清欠页查看账龄；逾期 90 天以上系统自动提醒客户负责人（修复后自动）。
4. **保函**：到期前系统自动提醒（修复后自动）。
**会收到的待办**：作为审批人（财务主管参与金额/毛利红线审批）；逾期回款/应收/保函等定时提醒。

### 7. 法务（legal）
**菜单**：合同管理（编辑+签署）；客户/商机/报价只读。
**日常**：审查合同条款（可用「AI 条款分析」）；作为合同审批人；审批通过后**签署合同**（合同详情「签署合同」）。签署后系统通知合同创建人，并解锁商机赢单。

### 8. 客户服务部 / 售后（service_engineer / service_manager）
**菜单**：售后工单、售后实测数据、工艺设备档案、续约管理；客户/合同/产品只读。
**日常**：建工单并指派 → **被指派人收到"工单分配"通知**；按 SLA 处理（系统按优先级自动算响应/解决时限）；记录实测数据；从到期合同/竞品设备发起续约机会。

### 9. 管理层（executive）
**菜单**：全模块只读 + 报表中心 + 操作日志（审计）。可作为高层审批人。看板/仪表盘查看全租户经营概览、风险预警、审批 SLA 概览。

### 10. 系统管理员（admin）
**菜单**：部门管理、用户管理、角色权限、操作日志、系统配置、API 文档、开放平台、系统健康、钉钉集成、数据管理。
**日常**：维护组织/用户/角色与数据范围；配置**审批策略 ApprovalPolicy**（决定报价/合同/变更/**方案**提交后谁来审批——没配就不会产生审批待办，提交人会收到"未配置审批流程"提醒，务必配置）；配置阶段闸门(StageDefinition)、通知模板、文件存储、钉钉/企微集成等。

## 五、本次随手修复的问题（"修复中间发生的问题"）

调研过程中发现并已修复以下与"待办/衔接"直接相关的缺陷：

1. **销售目标接口对所有人（含管理员）403**：`dashboard:view` 权限被接口校验但从未在种子里创建。已在 `scripts/seed.py` 补该权限，并授予销售专员/主管/总监/高管；管理员自动拥有全部权限。（部署会自动跑 seed 生效）
2. **逾期类提醒从不自动发出**：回款已逾期、应收逾期 90 天、保函到期三类检查此前未接入定时 worker，只能手动点 API。已接入 `reminder_worker.run_once()`，逐租户自动执行。
3. **任务分配不通知被分配人**：派任务给他人时对方收不到任何提醒。已新增「任务分配」通知（单条/批量汇总）。
4. **线索/客户转交不通知新负责人**：销售交接时新负责人不知情。已为线索改派（单条+批量）和客户转交新增「线索分配/客户分配」通知。
5. **通知类型前端映射不全**：多种实际会发出的类型（提及、应收逾期、保函到期、定时报表、审批超时等）在前端会退化成"系统"图标。已补全铃铛与通知中心的类型映射。

### 第二轮已修复（原"已知限制"中的设计性问题）
- ✅ **通知偏好开关现已生效**：`send_notification` 会读取用户的 `NotificationPreference`，关闭的类型不再下发（采用 opt-out：未设置/未列出的类型默认仍发）。
- ✅ **方案审批接入审批引擎**：方案提交评审（状态→"评审中"）会自动发起审批流（biz_type `solution`，需在「系统配置→审批策略」配置方案审批策略）；审批通过后由 `_on_approval_completed` 自动把方案置为"已批准"，满足 S4 闸门。未配置策略时仍可手动批准作为兜底。
- ✅ **审批策略缺失不再静默**：报价/合同/变更/方案提交时若无匹配 `ApprovalPolicy`，会给提交人发一条通知（"已提交，但未配置审批流程，请联系管理员"），不再无声卡在"已提交"。

### 仍存在的已知限制（建议后续处理，本册如实标注）
- **线索→商机无自动转化**：转化只生成客户，需求/预算等上下文不会带入商机，需销售手动建商机。（产品决策，暂保持现状）
- **闸门只校验"存在"不校验状态**：S5/S6 只要求"有报价/有合同"，草稿状态也算；真正的成交红线是"赢单需已签署合同"。可按租户在阶段定义(StageDefinition)中自定义更严格的闸门，故未强行收紧默认行为。

## 六、给各部门的一句话总结

- **销售**：录线索→转客户→建商机→逐阶段推进（卡住看闸门提示）→提交报价触发审批。
- **售前**：在指派的商机里出方案 → 提交评审（触发方案审批）→ 审批通过自动置"已批准"，否则销售卡在 S4。
- **商务**：维护标书、由报价生成合同。
- **法务**：审合同→审批→签署（签署才能赢单）。
- **财务**：合同付款条款一键生成回款计划→登记到账自动对账；关注逾期自动提醒。
- **生产/采购**：赢单后建交付里程碑、维护订单。
- **售后**：建并指派工单（对方有通知），按 SLA 处理。
- **管理员**：先把"审批策略"配好，否则审批待办根本不产生。
- **所有人**：待办分散在【铃铛 / 审批中心 / 待办任务 / 工作台卡片】四处，分别查看。
