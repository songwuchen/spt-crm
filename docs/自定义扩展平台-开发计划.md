# SPT-CRM 自定义扩展能力 — 评估与开发计划

> 目标：让 CRM 适应更多企业的差异化需求，新增四大能力：**① 各类表单自定义字段、② 自定义表单、③ 可视化自定义审批流程、④ 可配置仪表盘**。
> 复用来源：`F:\dev\spt-lowcode`（一套已成熟、对标简道云的低代码/OA 平台）。

---

## 一、现状评估

| 能力 | CRM 现状 | 结论 |
|---|---|---|
| **自定义字段** | 已有 `custom_field_defs` 表 + 业务表内联 `custom_fields_json` 列；`CustomFieldsPanel.tsx` 渲染。**仅 `customer`、`project(商机)` 真正落库**；lead/contact/service_ticket 允许定义但**无存储列**。仅 6 种字段类型（text/number/date/select/multiselect/boolean），无校验、无联动、无字段权限。 | **可用但不完整**，可增强复用 |
| **自定义表单** | **无**。所有业务表单为硬编码 antd Form，无表单引擎、无 schema 驱动渲染。 | **需从 lowcode 移植** |
| **可视化审批流程** | 有 `ApprovalPolicy`（`condition_json` + `approver_rules_json`）+ `ApprovalFlow/Task` 顺序/并行/或签执行；`ApprovalPolicyModal` 只有**线性预览**（手绘 div），**非可编辑 DAG**。无条件分支路由、无节点图。 | **需移植可视化流程引擎** |
| **仪表盘** | 固定卡片目录 + localStorage 显隐/排序；`@ant-design/charts`。**无网格布局、无拖拽、无服务端可配置仪表盘**。 | **需移植仪表盘设计器** |

### lowcode 可复用资产（对标简道云，已跑通）

**后端（JSONB schema 驱动，易移植，无动态建表/无 EAV）**
- 表单引擎：`form_template` + `form_template_version`（`field_definitions`/`layout_definition`/`rule_definitions` 三个 JSONB）+ `form_instance`（`form_data` JSONB + 提交时 `field_definitions` 快照，保证历史可渲染）。
- 字段类型：`app/core/constants.py::FieldType`（~30 种，含 person/department/detail_table 子表/formula/auto_number/signature/address 等）。
- 流程引擎：`wf_process_definition(+version)`（`node_definitions`/`route_definitions`/`approver_rules` JSONB）+ 运行时 `wf_process_instance`/`wf_node_instance`/`wf_task_instance`/`wf_task_action_log`；引擎在 `services/workflow/`（engine/task_service/approver_resolver/field_permission）。节点类型 start/approval/cc/condition/parallel/merge/webhook/sub_process/end；审批人类型 10+ 种；分支挂在 `route_definitions.condition` 边上。
- 仪表盘：`Dashboard`（`components` JSONB）+ `services/dashboard/aggregation_service.py::AggregationService`（**运行时对 `form_instance.form_data` JSONB 直接 GROUP BY 聚合**，含公式解析器/子表数组展开/人员部门/日期粒度/地址下钻/数据权限）。

**前端（4 个设计器，均为「左调色板 | 中画布 | 右属性」三栏 + Zustand store + 撤销重做）**
- 表单设计器 `pages/admin/form-designer/`（~7,466 行；最大文件 `PropertyPanel.tsx` 2,414 行）。
- 表单渲染器 `components/FormRenderer/`（~5,368 行；`index.tsx` 686 行内含公式/联动/前置事件/规则引擎运行时）。**编辑/只读/设计三态同一组件**；`FieldRegistry.ts` 做 `type→widget` 映射。
- 流程设计器 `pages/admin/workflow-designer/`（~4,313 行；`@xyflow/react` v12 + `@dagrejs/dagre` 自动布局；`NodeConfigPanel.tsx` 1,199 行）。
- 仪表盘设计器 `pages/admin/dashboard/`（~10,211 行；`react-grid-layout` 24 列 + `echarts`；`ChartEditor.tsx` 1,044 行）。
- （附赠）详情页设计器 `pages/admin/detail-designer/` — 自定义记录详情布局。

### 关键依赖差异（需新增到 CRM，均兼容 React 18）
| 用途 | 库 | CRM 现状 |
|---|---|---|
| 画布内排序拖拽 | `@dnd-kit/core|sortable|utilities` | 无 |
| 流程节点画布 | `@xyflow/react` v12 + `@dagrejs/dagre` | 无 |
| 图表 | `echarts` + `echarts-for-react` | 有 `@ant-design/charts`（并存或替换） |
| 仪表盘网格 | `react-grid-layout` | 无 |
| 富文本 | `react-quill-new` | 无（可选） |

## 二、总体架构决策

**在 CRM 内新增一个「扩展平台 / 自定义应用」模块（后端 `app/domains/lowcode/*`，前端 `pages/lowcode/*`），与现有硬编码业务域并存**，复用 CRM 既有基础设施：

- 复用：租户隔离（`TenantScopedBase` + tenant_id）、认证/当前用户、RBAC 权限（`resource:action`）、组织架构（`Department`/`UserDepartment`）、文件存储（可插拔本地/MinIO/OSS）、通知/outbox worker、审计。
- 不复用/不照搬 lowcode 的独立 auth/org/file/tenant 实现 —— 只移植**引擎与设计器**，接到 CRM 的基础设施上。

四大能力落位：
1. **自定义字段** → 增强现有轻量系统（不走表单引擎），但**复用移植后的字段 widget** 扩充类型。
2. **自定义表单** → 新模块的表单引擎（template/version/instance + FormRenderer + form-designer）。
3. **可视化审批** → 新模块的流程引擎（wf_* + workflow-designer），**先服务自定义表单**，再桥接既有业务审批。
4. **仪表盘** → 新模块的仪表盘设计器 + AggregationService，先服务自定义表单数据，再接入 CRM 业务域数据源。

### 组织模型 gap（影响审批人解析）
CRM 有：部门树（`Department.parent_id/path`）、部门负责人（单 `leader_id`）、多部门（`UserDepartment`）。
lowcode 额外有：岗位(post)、多级上级链(leader)、代理审批(agent)。
→ 移植 `approver_resolver` 时，**指定用户/指定角色/部门负责人/表单字段选人** 可直接适配 CRM；**多级上级链/岗位/代理** 需在 Phase 0 决定「扩展 CRM 组织模型」或「先降级不支持」。

## 三、分阶段路线图（垂直切片，逐段可用、段间评审）

### Phase 0 — 地基（前置，约 1 段）
- 新增前端依赖（dnd-kit / xyflow / dagre / echarts / echarts-for-react / react-grid-layout）并验证 React 18 构建通过。
- 后端建 `app/domains/lowcode/` 骨架 + Alembic 迁移：`form_template`、`form_template_version`、`form_instance`（tenant_id 隔离）。移植 `FieldType` 枚举 + `FieldDefinition` schema。
- 权限位：新增 `form:*`/`workflow:*`/`dashboard:*` 权限并写入 `scripts/seed.py`（否则 admin 403）。
- 组织适配层：实现 `approver_resolver` 到 CRM `Department/UserDepartment` 的映射；确定多级上级/岗位/代理的取舍。
- 菜单/路由：新增「扩展平台」一级菜单占位。

### Phase 1 — 自定义表单引擎 MVP（能力②）
- 前端移植 `FormRenderer` + 一线字段子集（text/textarea/number/amount/date/datetime/select/multi_select/radio/checkbox/switch/person/person_multi/department/file/image/detail_table 子表/formula/auto_number ≈ 20 种）+ 规则引擎（显隐/只读/必填/公式）。
- 前端移植 `form-designer`（palette/canvas/PropertyPanel/表单属性）。
- 后端：模板 CRUD + 版本化 + 发布；实例 创建/查询/列表/详情/编辑/删除；公式服务端计算 + 自动编号。
- **验收**：管理员拖拽建表单→发布；员工填报→提交→列表/详情查看。**「自定义表单」达成**。

### Phase 2 — 自定义字段增强（能力①，依赖 Phase 1 的 widget）
- 复用移植后的字段 widget 升级 `CustomFieldsPanel`：扩类型（日期时间/人员/部门/金额/多选/附件等）+ 值校验 + 简单显隐联动。
- 为所有目标实体补 `custom_fields_json` 列（lead/contact/service_ticket/order/contract…）+ Create/Update/Out schema 白名单 + 详情/表单页挂载面板 + 放开 `entity_type` 正则。
- 服务端按字段定义做必填/类型校验。

### Phase 3 — 可视化审批流程（能力③）
- 前端移植 `workflow-designer`（xyflow 画布 + NodeConfigPanel 审批人/字段权限/节点操作/流转规则 + 条件分支边）。
- 后端移植 `wf_*` 引擎：定义/版本/发布；运行时 提交/通过/驳回/撤回/转审/加签/抄送/催办/评论；待办任务实例；SLA 超时；审批人解析（接 CRM 组织）；事件走 CRM outbox + 通知/钉钉。
- 先绑定自定义表单；**可选**：为既有业务审批（quote/contract/order/lead…）提供「可视化 DAG」——评估是「新引擎替换旧 ApprovalPolicy」还是「并存桥接」（旧引擎风险高、已接生产，倾向并存/渐进迁移）。

### Phase 4 — 可配置仪表盘（能力④）
- 前端移植 `dashboard-designer`（react-grid-layout + echarts + ChartEditor/ConfigPanel）+ 组件渲染器（柱/条/线/饼/漏斗/雷达/指标卡/透视表/明细表/地图…）。
- 后端移植 `AggregationService`（对 `form_instance.form_data` 运行时聚合）+ 聚合/明细/地图数据接口。
- 数据源：先自定义表单；再新增「CRM 业务域数据源」适配器（对真实业务表聚合），让仪表盘也能图表化现有 CRM 数据。
- 与现有 `Dashboard.tsx` 工作台并存/融合策略。

### Phase 5 — 打磨与集成
- 移动端渲染、字段级/数据级权限、模板市场（预置模板一键启用）、审计留痕、导出（Excel/PDF）、性能（JSONB 索引 + 热点冗余列 + 缓存）。

## 四、风险与权衡

- **工作量大**：lowcode 设计器前端 ~26k 行 + 大量后端引擎，属多段迭代工程，非一次性交付。→ 垂直切片、逐段验收。
- **React 18 vs 19 / 无 Tailwind**：需逐组件校验 hooks/新 API 与样式改造。
- **既有审批引擎迁移风险**：`ApprovalPolicy`+`ApprovalFlow/Task` 已接生产多业务域，直接替换为 DAG 风险高。→ 新引擎先服务自定义表单，旧引擎并存，后续渐进迁移。
- **组织模型 gap**：多级上级链/岗位/代理审批需扩展 CRM 组织模型或先降级。
- **测试库污染**：后端 pytest 直连生产 DATABASE_URL（见记忆），新迁移/测试需在干净库验证。
- **生产部署**：push→CI（测试 + 构建推 Harbor）；生产机手动 `docker compose pull` / 迁移 / 升级（见 `deploy/deploy.sh`）；prod 无 Docker Hub（compose pull 需排除或走私有镜像）。

## 五、已确认决策（2026-07-14）
1. **起步顺序**：✅ **自定义表单引擎优先**（foundation-first，一切统一到它）。
2. **可视化审批范围**：✅ **替换旧审批引擎**——用新 DAG 引擎统一替换 `ApprovalPolicy`+`ApprovalFlow/Task`；需数据迁移与既有 biz_type（quote/contract/change/order/lead/service/solution）回归。
3. **自定义字段架构**：✅ **统一到表单引擎**——现有硬编码业务表单逐步改为 `FormRenderer` 按 schema 渲染；自定义字段即表单字段。
4. **组织模型**：✅ **本期扩展**——新增 岗位(post)/多级上级链(leader)/代理审批(agent)，支持全部审批人类型。

### 由决策推导的修订路线
- **表单引擎是内核**：所有业务实体最终以「系统内置表单模板（受保护/部分字段硬绑定）+ 自定义扩展字段」的统一模型存在；迁移分批、与旧硬编码页面并存过渡。
- **审批替换是重活**：新 `wf_*` 引擎上线后，为每个既有 biz_type 写「触发+回写」适配，灰度切换，保留回滚。
- **组织模型扩展进 Phase 0/3**：post/leader/agent 表 + 管理 UI，`approver_resolver` 全类型支持。
