## 二、关键设计与映射规则

1. **主数据复用，不重导**：生产已存在客户 16,285 / 用户 726 / 部门 125（行业已映射为 `screening_*` 代码）。迁移仅做**按名称关联**，不重复导入客户/用户/部门。
2. **项目中心化改造**：SPT‑CRM 的合同/报价/回款/发票均要求 `project_id`，而简道云以合同为中心。故**每张合同合成一个商机项目**（S6/won），把合同、回款、发票挂到该项目下。
   - 连接键：`合同号` 与 `图纸编号` 写入辅助表 `_contract_join`，供收款/开票按键回挂。
3. **状态/阶段映射（申报信息→商机）**：已签合同→won/S6；中标→won/S5；进行中→active/S2；暂停→suspended；取消/落标/流标→lost。
4. **负责人/部门**：按简道云 `业务员/申报人` 的姓名匹配生产 `users.real_name`；匹配不到则保留 `owner_name` 文本、`owner_id` 置空。
5. **客户关联**：商机 `公司名称` 命中主数据则关联，未命中**不污染主库**，公司名暂存 `custom_fields_json.公司名称`（约 60% 商机未命中，多为历史/落标项目）。合同链客户未命中则补建（仅 ~5%）。
6. **来源留痕**：所有迁移行 `custom_fields_json._src/_src_id` 标记来源；并建辅助表 `_migration_map(source_form, source_id, target_table, target_id)` 保证幂等与可追溯。

## 三、幂等与可重跑

- 脚本支持按阶段重跑：`python jdy_etl.py <projects|contracts|payments|invoices|service|activities|counts> [limit]`。
- 已迁移行通过 `_migration_map` 跳过，**重跑只补增量**（本次 activities 因一次瞬时短页中断，修复分页后重跑自动补齐缺失的 ~1.2 万条）。
- 分页改为「按统计总数遍历窗口 + 每窗重试」，避免瞬时短页导致提前结束。

## 四、已知差异与未迁移项（需业务确认/后续处理）

1. **未匹配客户的商机（~60%，21,774 条）**：已执行 relink 数据治理（`jdy_etl.py relink-preview/relink-apply`）。按公司名精确匹配仅 **615 条（2.8%）** 能挂接到现有客户主数据（已挂接）；其余 **21,159 条** 公司不在主数据中（多为休眠/历史报备潜在客户），经决策**保持现状**：公司名保留在 `custom_fields_json.公司名称`，商机列表仍可见，不为其批量新建客户以保持主数据干净。如后续需要，可人工筛选后再转客户。
2. **无客户的销售日志（14,451 条）**：纯工作汇报、无客户拜访对象，未建 activity（activity 必须挂 biz）。如需保留，可改挂到「填写人」维度的日报模块。
3. **回款无合同匹配（1,432 笔）**：对应合同不在「合同登记表」（更早合同/现金流科目类），未建回款记录。
4. **商机↔合同未打通**：源系统两者无可靠 join key，迁移后为两套独立项目（申报漏斗项目 + 合同项目）。建议后续按「客户+名称」做合并/挂接。
5. **未纳入本次范围**：业务提成/奖金、应收清欠/抢单、保函/保证金、精细化营销工艺档案、现场设备实测数据等——见《SPT‑CRM待实现功能开发计划》。

## 五、回滚说明

所有迁移数据均可定位与清理（按 `_migration_map` 与 `custom_fields_json._src` 标记）。如需回滚某阶段：
```sql
-- 例：回滚发票
delete from invoices where id in (select target_id from _migration_map where target_table='invoices');
delete from _migration_map where target_table='invoices';
```
补建的客户：`delete from customers where custom_fields_json::text like '%jdy_migration%';`（注意先处理其下引用）。
