# 从简道云到 SPT‑CRM：CRM 业务逻辑梳理与习惯改变指南

> 调研对象：低代码平台「AI 数据接口生成器」所连接的简道云（JianDaoYun）数据中心
> 调研方式：通过平台开放接口（`/api/app/list`、`/api/app/{id}/entries`、`/api/form/{app}/{form}/fields`、`/api/form/list`、`/api/form/statistics`）逐表读取结构、字段与数据量
> 目标系统：SPT‑CRM（FastAPI + SQLAlchemy + React，本仓库）
> 编写日期：2026‑06‑05

---

## 〇、结论速览（给管理者）

1. 现有「CRM」并不是一个独立系统，而是**散落在简道云 30+ 个应用、200+ 张流程表单里的一组业务流程**。核心集中在三个应用：**销售中心**（前端业务）、**数据中心**（合同/回款/客户主数据/提成）、**客户服务部**（售后）。
2. 简道云的运转逻辑是「**一件事 = 一张流程表单 + 一套审批节点**」。优点是灵活、谁都能加表；代价是**表单爆炸、字段冗余、跨表靠手工关联、大量辅助/防重/校验字段、历史停用表堆积**。
3. SPT‑CRM 的逻辑是「**对象（客户/商机/报价/合同/回款/工单）+ 阶段门禁 + 配置化审批 + 自动级联**」。同样的业务用**固定结构 + 配置**承载，而不是不断新建表单。
4. 迁移的本质不是「换个录入界面」，而是**思维方式从「表单流程」转为「对象+阶段」**。本文第五章给出按角色的习惯改变清单，第六章给出功能缺口与迁移建议。

## 一、源系统画像

### 1.1 平台与应用
- 平台：**简道云**（图标资源域名 `jdycdn.com`、字段命名 `_widget_*`、字段类型 `sn/subform/linkfield/linkquery/lookup/usergroup/dept` 等均为简道云特征）。
- 共 **34 个应用**。与 CRM 强相关的：

| 应用 | 定位 | 与 CRM 的关系 |
|---|---|---|
| **销售中心** | 前端销售作业 | 项目报备、销售日志、合同/技术评审、售前服务、发货通知 |
| **数据中心** | 公司运营中枢（最大） | 客户主数据、合同登记、收款、开票、应收账款/清欠、业务奖金/提成 |
| **客户服务部** | 售后 | 服务申请、服务记录、服务评价、客服日志 |
| 国际营销中心 | 外贸 | 多为用车/用章等已停用流程，CRM 价值低 |

> 数据中心同时承载了大量**非 CRM**业务（借/付款、保函保证金、费用报销、奖罚、在建工程、出口退税、批款等），这些不在迁移范围，但说明了「数据中心 = 公司全业务表单仓库」。

## 二、简道云中的 CRM 业务逻辑（端到端梳理）

下面按真实业务流向，逐环节说明「简道云怎么干的」「关键字段」「隐含规则」。

### 2.1 客户主数据 —「客户信息」（16,408 条）
- 关键字段：客户编号(sn)、客户名称、注册资金/实缴资本、成立年份、地址、所属行业(工业升级/循环经济/基建民生/技术改造)、客户性质(民企/国企控股/国企/上市)、客户关系(直接/间接)、**客户类型 A/B/C/D**、主联系人职位、工资及保险情况(正常/有拖欠/非正常)、**联系人明细(子表单)**、外贸客户信息(国别/客户代码/关注产品/客户来源/邮箱/主页)、开票信息(纳税人识别号/开户行账号)、智能化备案信息（法人/员工数/年运行天数/占地/年用电量…）。
- 隐含规则 / 痛点：
  - **一张表塞了三类客户**（内贸、外贸、智能化备案），靠"分隔符"分区，字段极多且很多场景用不到。
  - **无强制查重**，16,408 条里存在同名/重复客户（靠人工辨别）。
  - 客户与业务员/部门的归属，靠 `业务员(user)`、`业务部门(dept)` 字段手填。

### 2.2 线索/商机漏斗 —「申报信息(项目报备)」（36,424 条）+「180 天项目激活」
- 关键字段：项目编号(sn)、**来源(自报/分发)**、项目名称、公司名称、客户类型(终端-央企国企/大型民企/一般民企/设计院/总包/配套贸易/其他)、项目地址、是否内部冲突、**行业(细到"筛分分选‑冶金/矿山/砂石/焦化…")**、**中标情况(中标/结果未出/取消/延期/落标/流标/未参与)**、原因(价格/内定/资质/技术/现场/质保/付款/工期…)、委托状态(已开/未开)、**项目动态(技术交流/出方案/报价/投标/拟建)**、**项目状态(进行中/暂停/取消/落标/中标/已签合同)**、项目最终状态(待审/收录/回退)。
- 业务逻辑：
  - 这是事实上的**销售漏斗 + 线索分发**：业务员"自报"或被"分发"项目 → 跟进 → 更新动态/状态 → 中标后转合同。
  - **「180 天项目激活」**：超过 180 天未推进的项目需走"激活"流程重新报备——即**陈旧商机唤醒机制**。
  - **「内部冲突」字段**用来人工判断撞单。
- 痛点：报备是**一次性表单**，后续动态靠改同一条记录或新建激活单；没有结构化阶段，状态全靠下拉枚举手选；漏斗分析靠仪表盘。

### 2.3 线索分发 / 公海 —「G 信息分发‑公海池」「G 跟进记录」（已关停）
- 早期存在"公海池 + 跟进记录"，现已关停（停用未用组）。客户信息表里仍保留 `信息分发(是/否)`、`信息来源(经理分配/情报部分配/自己获取)` 字段，说明**线索分配的诉求一直存在**。

### 2.4 日常跟进 —「销售日志」（37,214 条）
- 关键字段：日志日期、所属区域/大区经理、**是否出差/是否催收**、今日完成/未完成/问题/学习心得/明日计划、**拜访明细(拜访时间、客户名称、跟进方式[当面/电话/网络]、跟进信息[签合同/招标/拓展/售后/催款/售前/其它]、详细描述、定位、图片、下次跟进时间)**、用车/行程子表单。
- 业务逻辑：日报 = 工作汇报 + 客户跟进记录 + 出差/用车，**三合一**。
- 痛点：
  - 表头有 **30+ 个"辅助‑大区经理 N / 编号辅助 N / 填写人是否等于大区经理 N"** 字段——这是为"按大区经理汇总/校验"硬造的辅助列。
  - "每人同一天只能交一次日志检验"是一个**文本字段 + 校验规则**做的防重。
  - 跟进记录埋在日志里，**没有以客户/商机为中心的时间线**。

### 2.5 售前服务派工 —「售前服务通知流程」（2,573 条）
- 关键字段：项目状态(已签/已中标/技术交流)、是否智能化、希望派遣人员、合同号、服务地点/时间/天数、**工作日程子表单**、**测绘数据子表单(名称/规格/数量/单位/单重)**、是否需要新疆威猛人员等协同。

### 2.6 签约门：合同/技术评审 —「合同评审」(11,419)、「合同技术协议评审」(993)
- 「合同评审」字段非常丰富，是**签约前的总闸门**：
  - 基本：合同评审/项目评审、是否出口、是否核价、是否需要安装、业务员/区域经理/业务部门、公司名称(linkfield 关联客户)、电控装置选项。
  - **核价信息**：关联核价单、成本价、配置要求（电机/轴承/主材/衬板）。
  - **客户信息**：客户类型(新/老)、公司性质、所属行业、规模资金、客户关系、失信/诉讼/环保/税务/其它处罚（5 项风险计数）、**联系信息子表单**。
  - **项目信息**：项目名称及应用、报备时间、合同价格、交货期、**是否有保函/保函类型**、**是否有重量要求**、**是否趁用呆滞设备**、是否含智能化、付款方式、现场测绘要求、附件。
  - **风险信息**：**法务/技术/业务/财务/采购/出口 6 个维度，各自"高/中/低"等级 + 文字描述**。
  - **结论**：账期、结论描述、是否反馈、合同签订（图纸编号/合同明细 linkquery）。
- 业务逻辑：这是一个**多部门会签的审批流程**（业务→各风险部门→结论），通过后才允许签约/登记合同。
- 痛点：客户、项目、报价、风险信息在评审表里**再录一遍**（与客户信息/报备表重复）；风险等级靠人工手选。

### 2.7 合同登记 —「合同登记表」（19,340 条）
- 关键字段：合同号、对方合同号、图纸编号、单位名称(combo)、客户编号(linkfield 关联客户)、部门/业务人员、合同状态(新增/变动)、合同获取方式(公开招标/邀请招标/协商一致)、**关联"合同/项目评审"(linkfield)**、**合同明细子表单(产品类型/名称/规格/数量/单价/总价/外币/电控/技术参数)**、合同总金额、**收款计划子表单(日期/付款方式/比例/金额/是否提醒)**、质保期限/金额、行业/地区/应用领域/应用物料、是否含智能化+智能点、运费承担方、**合同/发货地址(并做"到区"比对辅助)**、验收方式/资料/上传、回款信息/开票信息(linkquery)、发货状态。
- 业务逻辑：合同登记 = **合同主档 + 明细 + 收款计划 + 验收要求 + 与评审/回款/发货/开票的关联枢纽**。
- 痛点：
  - 大量**辅助/防重字段**："限制同一合同号只能新增一次"、"判断是否含新增"、"辅助是否含需要提醒"、地址"到区"比对辅助等。
  - 与回款/发货/开票全靠 `linkquery` **手工建立查询关系**。
  - "新增/变动"靠状态字段表达版本，**没有真正的版本化**。

### 2.8 发货 —「发货通知流程」（23,477 条）
- 关键字段：合同号(linkfield)、合同金额/累计已发/未发、发货类型(公司内发/采购直发)、收货单位/人/电话、要求到货时间、多地址、运费承担、**发货明细子表单**、验收单附件、回执单等。

### 2.9 开票 —「开票申请」（11,999 条）
- 关键字段：图纸编号(linkfield)、单位名称、合同数据(linkfield)、**合同明细(新增/变动)子表单**、总价合计(调整后)、开票时间、发票号码、接收邮箱。

### 2.10 收款 —「收款登记流程」（24,745 条）
- 关键字段：收款号、来款日期、单位名称、**来款明细子表单(来款形式/现金流/科目代码/银行/金额/承兑号/到期日)**、**款项分配子表单(图纸编号/合同号/款项性质/分配金额)**、当日回款总数/目标数。
- 业务逻辑：到款先登记，再**分配到具体合同/图纸**，支撑应收与提成。

### 2.11 应收与清欠 —「应收账款流程」(4,391)、「应收清欠流程」、「客户转换部门」(283)、「销售经理抢单」(27)
- 应收账款流程：**每月每客户每业务员一条**（防重字段保证），记期初/期末欠款、本月发货/回款明细、未到期/已到期应收、客户付款计划。
- 清欠链路：**客户转换部门**（销售直接转清欠/清欠转回销售/转法务/部门间转移/财务转诉讼）→ **销售经理抢单**（被移交客户由其他销售"接单"承接清欠）→ 应收清欠裁决/清收函件审批。
- 业务逻辑：这是一个**欠款客户的责任重新分配 + 催收**机制，带考核日期与承诺。

### 2.12 业务提成 —「业务奖金流转单」(11,328)、「提成数据库」(11,284)
- 关键字段：奖金编号、图纸编号、业务员/部门、签订日期、合同明细、合同金额、**来款明细(关联收款)**、结算状态(%)、运费/服务费/招待费/返还款、已提比例/已提奖金、本次奖金金额、**支付状态子表单**、总经理审批。
- 业务逻辑：**回款驱动的提成核算**——按合同回款进度，扣减运费/服务费/招待费/返还后，按比例计提并分次支付。

### 2.13 售后服务 —「客户服务申请及反馈」(6,561)、「客户服务记录」(2,853)、「客户服务评价」(1,820)、「客服日志」(17,777)
- 服务申请及反馈：业务员发起，**服务性质(售前测绘/售中安装调试/售后维修)**、紧急程度、客户种类(战略/大/一般)、**主要产品信息子表单(合同号/设备名称/型号/数量/发货日期)**、转交/协作/通知人员、是否需总经理批示。
- 服务记录：现场服务结果，含**大量设备实测数据**（分级粒度、振幅、电机功率、料层厚度、入料粒度、筛分效率、处理量、振源温度、运行电流…）、筛芯/筛板子表单、客户签字、对服务人员/产品的评价。
- 服务评价：合同号、单位、业务评分/评价。
- 客服日志：现场日志 + 费用子表单 + 加班登记。
- 业务逻辑：**工单制 + 现场技术记录 + 满意度评价**，且与合同号强关联。

### 2.14 精细化营销 —「精细化营销推进流程」（737 条）
- 关键字段：客户名称、行业、主要产品、年产量、下属分厂、客户工艺文字描述、**按行业拆的多套工艺子表单(冶金/焦化/电力/矿山/煤炭/砂石/精细物料)**、备件使用情况、设备使用情况。
- 业务逻辑：对重点客户做**深度工艺/设备调研**，为后续方案/替换销售铺垫。

### 2.15 跨表协作的两大"简道云特征"
1. **流程表单 + 节点审批**：几乎每张表都是"流程"，数据里带 `flowInstanceId`、`_optAuth`(节点字段权限)、`_dataPerms`(数据权限)。业务推进 = 表单在审批节点间流转。
2. **手工跨表关联**：`linkfield`(选择另一张表的记录) + `linkquery`(查看关联表数据) + `lookup`(取值)。例如合同登记里"选择合同/项目评审信息""客户编号查询""回款信息查询"全是手工建立的关联。

## 三、SPT‑CRM 的对应业务模型（目标系统）

SPT‑CRM 用**领域对象 + 阶段 + 配置化流程**承载上述业务：

| 领域对象 | 表 | 能力要点 |
|---|---|---|

| 过去 | 现在 |

### 5.6 全员通用的 7 个思维转变
1. **从"表单流程"到"对象+阶段"**：不再"为一件事建一张表"，而是围绕客户/商机/合同等固定对象推进。
2. **从"流水号/手工编号"到系统编号**：`project_code/quote_no/contract_no` 由系统生成。
3. **从"节点字段权限(_optAuth/_dataPerms)"到"角色权限 + 数据共享(AclShare) + 公海"**：权限是配置出来的，不是埋在每个流程节点里。
4. **从"linkfield/linkquery 手工关联"到"自动级联"**：客户→商机→方案→报价→合同→回款→发票天然串联。
5. **从"复制表单做新流程"到"配置阶段门禁/审批策略"**：变化用配置承接。
6. **从"状态字段表达版本"到"真正的版本化"**：报价/合同/方案均有版本与审批留痕。
7. **从"人造辅助/防重/校验列"到"系统内置约束与校验"**：不再需要"是否等于大区经理 N""限制只能新增一次"这类字段。

## 六、功能缺口与迁移注意事项

### 6.1 SPT‑CRM 目前**没有**直接对应、需决策的能力
| 简道云能力 | 现状 | 建议 |
|---|---|---|
| **业务奖金 / 提成核算**（流转单 + 提成台账，11k+ 条） | 无对应模块 | 短期：提成继续在原系统/ERP 算；中期：基于回款记录新增"提成"模块（按合同回款进度计提） |
| **应收清欠"抢单"机制**（移交→抢单→承诺/考核） | 仅有负责人变更/公海领取 | 若清欠考核重要，需把"抢单+承诺+考核日期"做成专门流程或审批 |
| **保函/保证金管理** | 商机仅有 `has_guarantee` 布尔 | 如需台账（到期提醒/退还），单列模块或用任务+附件管理 |
| **出口订舱/报关/退税、批款、费用报销、商务接待、礼品/收据** | 非 CRM 范畴 | 留在简道云/OA/ERP，不迁入 CRM |
| **精细化营销的多行业工艺子表单** | 无专表 | 作为方案输入或客户 `custom_fields_json` 承载 |
| **现场设备实测数据**（筛分效率/振幅/处理量等强专业字段） | 工单无专用结构 | 用工单自定义字段或扩展工单子表 |

### 6.2 数据迁移建议
1. **客户去重优先**：16,408 条客户存在大量重复/外贸/智能化备案混录，迁移前按"名称+纳税人识别号"做去重与合并，建立**主客户档案**。
2. **报备→线索/商机的拆分**：36,424 条报备中，未中标/早期的转为**线索**，进行中/已中标的转为**商机**并落到合适阶段；中标记录可生成 **Tender(已中标)**。
3. **合同与回款历史**：合同登记(19,340)→合同 + 明细 + 收款计划；收款登记(24,745)→回款记录并按合同/款项分配回填，保证应收能自动算账龄。
4. **关联还原**：把简道云里 linkfield/linkquery 的手工关联，转成 SPT‑CRM 的**外键级联**（客户↔商机↔报价↔合同↔回款↔发票）。
5. **历史"停用/关停/导删"表单不迁**：数据中心"停用未用"组体量巨大，仅做归档备查。
6. **枚举对齐**：客户类型、行业、来源、中标原因等下拉，统一进 **DataDictionary**，避免口径分裂。

### 6.3 迁移收益与代价（坦白说）
- **收益**：消除表单爆炸与重复录入；客户/商机为中心的视图与漏斗；自动级联与账龄；配置化审批与权限；分析与 AI 能力。
- **代价 / 风险**：
  - 简道云"想加就加一张表"的**极致灵活**会被结构化约束替代，部分边缘流程需走配置或暂留原系统；
  - 提成、清欠抢单、保函等**特定能力存在缺口**，需排期或并行；
  - 用户**习惯迁移成本**（尤其业务员从"写日志"转向"记跟进+推阶段"），需培训与样例。

## 七、落地建议（分阶段）

1. **第一阶段（主干跑通）**：客户/联系人 + 线索/商机(S1–S6) + 跟进 Activity + 报价/合同 + 回款/发票。覆盖报备、日志、评审、合同、回款、开票六大高频环节。
2. **第二阶段（售后与协同）**：服务工单 + 满意度 + 续购；交付里程碑；审批中心(报价/合同/方案/变更)；客户公海与分配。
3. **第三阶段（补缺口）**：提成核算模块、清欠抢单流程（如需）、保函/保证金台账（如需）、精细化营销/工艺调研扩展字段。
4. **贯穿全程**：数据迁移与去重、枚举字典统一、按角色培训、并行试运行后再停用对应简道云表单。
