## 1. 业务提成 / 奖金核算（P0）

**简道云现状**：「业务奖金流转单」按合同回款进度，扣减运费/服务费/招待费/返还款后，按比例计提奖金，分次支付；「提成数据库」为台账。核心逻辑：**提成 = f(合同金额、累计回款、扣减项、提成比例、已提金额)**，回款驱动、分次结算。

**SPT‑CRM 现状**：无对应模块（领域清单中无 commission）。已有合同金额与回款记录，可作为计算输入。

**建议方案**：新增 `commission` 领域。
- 模型：
  - `CommissionRule`（提成政策）：适用范围(部门/产品类型/客户等级)、比例、阶梯、扣减项配置。
  - `CommissionRecord`（提成单）：`project_id/contract_id`、业务员、合同金额、累计回款、扣减项(运费/服务费/招待费/返还)、应计奖金、已提比例、本次金额、状态(草稿/审批中/已核准/已支付)。
  - `CommissionPayout`（支付明细）：分次支付时间/金额/方式。
- 计算服务：监听回款记录变化 → 重算结算比例 → 生成/更新提成单（含审批，复用现有 Approval）。
- 关联：合同(amount_total) + 回款(payment_records 累计) 自动驱动。
- 报表：业务员维度提成台账、支付情况看板。
- 迁移：把简道云历史「提成数据库」导入 `CommissionRecord`（已迁移合同/回款可作关联）。

**验收**：录入合同+回款后能自动算出应计/可提奖金；提成单走审批与分次支付；与历史台账对账一致。

## 2. 应收账款账龄与清欠流程（P0）

**简道云现状**：
- 「应收账款流程」每月每客户每业务员一条，记期初/期末欠款、本月发货/回款、未到期/已到期应收、客户付款计划。
- 清欠链：「客户转换部门」(销售转清欠/转法务/部门间移交/财务转诉讼) →「销售经理抢单」(被移交客户由其他销售接单承接) → 裁决/清收函件审批，带考核日期与承诺。

**SPT‑CRM 现状**：有 `PaymentPlan/PaymentRecord`，可算应收，但**无账龄视图、无清欠/移交/抢单流程**。

**建议方案**：
- **账龄引擎**（P0）：基于 `payment_plans`(应收) − `payment_records`(已收) 自动计算每客户/项目应收余额与账龄分桶(0–30/31–60/61–90/90+/180+)；提供应收看板与预警（复用 reminder_worker）。
- **清欠/责任移交流程**（P0）：新增 `collection` 领域或扩展客户/项目：
  - `DebtTransfer`（客户转换部门）：转移类型(转清欠/清欠转回/转法务/部门间/财务转诉讼)、原/新部门与负责人、欠款余额、材料清单、考核日期、承诺，走 Approval。
  - **抢单**：转移发布后进入「待接单」池，目标部门销售在期限内 claim（参考线索公海/`batch_assign` 机制），记录接收人/承诺/考核日期。
  - `CollectionFollowUp`（催收跟进）：催收时间/形式(电话/现场)/客户反馈/预计收款时间，挂在客户或应收记录上（可复用 Activity）。
- 迁移：历史「客户转换部门/抢单」导入对应表，账龄由已迁移回款自动重算。

**验收**：应收账款看板按账龄分桶并预警；客户可发起移交→审批→目标部门抢单接收；催收记录可追溯。

## 3. 保函 / 保证金台账（P1）

**简道云现状**：合同评审有「是否有保函/保函类型(履约/预付/质量)」；数据中心有保函、保证金管理表与到期提醒。

**SPT‑CRM 现状**：商机仅 `has_guarantee` 布尔，无台账与到期提醒。

**建议方案**：新增 `guarantee` 领域：
- `Guarantee`（保函/保证金）：`contract_id/project_id`、类型(履约/预付/质量/投标保证金)、金额、出具机构、生效/到期日、状态(生效/已退还/已逾期)、附件。
- 到期提醒接入 `reminder_worker`（到期前 N 天通知责任人/财务）。
- 看板：在保/即将到期/应退未退。

**验收**：保函可登记并在到期前自动提醒；可追踪退还状态。

## 4. 精细化营销 / 客户工艺档案（P1）

**简道云现状**：「精细化营销推进流程」对重点客户做工艺/设备深度调研——按行业(冶金/焦化/电力/矿山/煤炭/砂石/精细物料)拆多套工艺子表单 + 备件使用 + 设备使用情况。

**SPT‑CRM 现状**：无专表；可临时塞 `custom_fields_json`。

**建议方案**：客户档案扩展「工艺/设备档案」标签页：
- `CustomerProcessProfile`：客户、行业、主要产品、年产量、分厂情况。
- `CustomerEquipment`（设备台账）：设备名称/规格/使用年限/供货厂家/现状（支持竞品替换识别）。
- `CustomerSpareUsage`（备件使用）：备件/更换频率/供货厂家/优缺点。
- 价值：作为方案设计与「替换销售/复购」的线索来源，可与 RenewalOpportunity 打通。

**验收**：可在客户下维护工艺/设备/备件档案；能据此筛选「使用竞品设备、临近更换」的客户生成复购线索。

## 5. 售后工单的现场设备实测数据（P1）

**简道云现状**：「客户服务记录」「客服日志」记录大量现场实测数据（分级粒度、振幅、电机功率、料层厚度、入料粒度、筛分效率、处理量、振源温度、运行电流、筛芯/筛板规格…）。

**SPT‑CRM 现状**：`ServiceTicket` 有 SLA/满意度，但无结构化现场技术数据。

**建议方案**：
- 工单增加「现场实测数据」子结构 `ServiceMeasurement`（设备型号/产品编号/各项实测指标/筛芯筛板配置/客户签字）。
- 沉淀为**设备运行数据库**，支撑设备健康分析、质量改进、复购预测。
- 满意度评价（客户服务评价）已可映射到 `satisfaction_score`，补充「业务评分/评价」字段。

**验收**：工单可录入并查询现场实测数据；可按设备型号聚合分析筛分效率/处理量等指标。

## 6. 发货 / 出门证管理（P2，部分已被里程碑覆盖）

**简道云现状**：「发货通知流程」(23,477) 含发货明细、累计已发/未发、收货信息、出门证、回执、运费承担。

**SPT‑CRM 现状**：有 `Order`、`DeliveryMilestone(ship)`，可承载发货里程碑，但无逐笔发货明细/出门证/回执。

**建议方案**（按需）：扩展 `delivery`：`Shipment`（发货单：合同/订单、收货单位/人、要求到货、出门证号、回执）+ `ShipmentLine`（发货明细）。若 ERP 已管发货，则只做轻量登记+里程碑联动。

**验收**：可登记发货单与明细，更新合同累计发货并联动交付里程碑。

## 7. 迁移收尾 / 数据治理（P0，与本次迁移配套）

本次已迁移：商机(申报信息)、合同链(合同+回款+发票)、售后工单、跟进(销售日志)。需要的收尾：
1. **客户去重**：16k+ 客户存在命名变体；建立「名称+纳税人识别号」去重与合并工具，合并时迁移其下商机/合同/工单。
2. **负责人补链**：迁移时 `owner_name` 已保留，但部分 `owner_id` 未匹配到用户（离职/同名）；提供按名批量匹配/手工指派工具。
3. **商机↔合同关联**：申报信息(商机) 与 合同登记(合同) 在源系统无可靠 join key，迁移后为两套独立项目；提供「按客户+名称」建议合并/挂接的工具。
4. **公司名称→客户挂接**：迁移的商机中约 60% 未匹配到主客户（公司名暂存 `custom_fields_json.公司名称`）；提供一键挂接/新建工具。
5. **枚举字典统一**：客户类型、行业、来源、中标原因等进 `DataDictionary`，对齐口径。
6. **回归校验**：与简道云仪表盘核对总数（合同总额、回款总额、商机数）确保一致。

**验收**：客户主数据去重率达标；迁移对象负责人匹配率 >90%；关键金额与源系统对账一致。
