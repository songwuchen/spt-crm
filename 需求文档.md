# 0. 文档控制

* 文档名：离散制造业 CRM + AI 全功能架构与开发文档（Rev.V1.0）
* 目标：支撑**线索→机会项目→方案→报价（多版本）→合同→订单交付→验收回款→售后复购**闭环
* 核心原则：

  1. **项目机会是主对象**（非“客户”）
  2. **版本化 + 变更链 + 可追溯**（报价/方案/条款/交期/成本都可回放）
  3. AI 必须**嵌入流程节点**，输出结构化结果，且**有依据、有权限、可审计**
  4. 与 ERP/MES/MOM **边界清晰**：CRM 管经营与决策快照，ERP 管执行与财务凭证

# 1. 总体架构

## 1.1 逻辑分层

1. **渠道接入层**：Web、移动端、企业微信/钉钉、邮件、表单、API

# 2. 业务域设计（核心对象与流程）

## 2.1 核心业务对象（强制版本与链路）

* Customer（客户）/ Contact（联系人）/ Account（客户组织）
* Lead（线索）
* OpportunityProject（机会项目）——主对象
* Solution（方案包）——可多版本，绑定附件/配置清单
* Quote（报价主单） + QuoteVersion（报价版本）——**版本化核心**
* Contract（合同） + ContractVersion（条款版本） + Approval（审批）
* OrderLink（订单映射）——对接 ERP 订单号
* DeliveryMilestone（交付里程碑）——生产/采购/发货/安装/验收状态回传
* Invoice / PaymentPlan / PaymentRecord（发票/回款计划/回款记录）
* ChangeRequest（变更单）——贯穿需求/报价/合同/交付
* ServiceTicket（售后工单） + SparePartQuote（备件报价） + Renewal（维保续费/复购）

## 2.2 阶段 Gate（可配置）

每个阶段有：**必填字段、必传附件、必走评审、AI 校验项**

* S1 线索：来源、行业、需求摘要、预算区间、决策链
* S2 立项：需求澄清表、现场条件、验收标准草案
* S3 方案：方案书、选型清单、风险点清单
* S4 报价：报价版本、成本快照、毛利红线校验、交期承诺校验
* S5 商务：条款要点、付款节点、关键异议与对策
* S6 成交：合同版本、订单映射、交付里程碑生成
* S7 交付：节点跟踪、变更闭环
* S8 回款：发票/回款计划、逾期预警
* S9 售后复购：维保、备件、改造机会

# 4. 权限模型（制造业必备：客户/项目/价格/附件分级）

## 4.1 RBAC + ABAC 混合

* RBAC：角色（销售、销售经理、技术、报价、财务、法务、售后、管理员）
* ABAC：策略属性（所属部门、项目owner、客户owner、密级、金额阈值、区域）

### 关键策略示例

1. **客户可见**：owner + 部门链 + 授权共享
2. **项目可见**：项目owner/协作人/参与部门
3. **价格字段脱敏**：无权限者只看区间/不显示成本与毛利
4. **附件密级**：restricted 仅项目授权+特定角色可下载
5. **AI RAG 检索权限继承**：检索只能命中用户有权限的文档（否则越权泄露）

## 4.2 表建议

* role、permission、role_permission
* user_role、user_department、department_tree
* acl_share（biz_type,biz_id,to_user/to_dept,rights）
* secrecy_policy（doc_type->默认密级）

# 5. AI 功能设计（可落地的“节点内嵌”）

## 5.1 AI 服务边界

AI 不直接改业务数据，只产生：

* **结构化建议**（result_json）
* **证据链**（evidence_json）
* **风险评级**（risk_level）
  由业务服务决定是否写入业务字段（例如把 AI 提取的需求字段“回填到表单”需人工确认或审批）。

## 5.2 核心 AI 任务清单（建议全部做）

1. **需求澄清提取**（lead/project）

* 输入：沟通记录/附件/表单
* 输出：需求字段、缺失项清单、追问清单、验收标准草案

2. **相似项目对比**（project/quote_version）

* 输出：相似度TopN、历史成交价区间、成本构成对比、风险点复用

3. **报价风险与毛利红线解释**（quote_version）

* 输出：风险条款/交期风险/技术风险/成本遗漏项/建议加价项，附依据

4. **合同条款风险**（contract_version）

* 输出：风险条款标签（罚则、验收、质保、交付、付款、知识产权）、修改建议、谈判话术要点

5. **会议纪要 + 下一步行动**（activity）

* 输出：决策人、异议点、行动项（负责人、DDL）、机会健康度

6. **智能推荐排序**（feed）

* 输出：机会优先级、跟进建议、可能复购客户列表（结合交付/售后/回款）

## 5.3 RAG 设计（知识库）

### 索引对象

* 产品手册、行业案例、技术标准、验收模板、报价规则、合同范本、售后知识、历史项目复盘
* 附件（PDF/Word/图片→需抽取文本；图片可多模态模型生成描述+标签）

### 权限与审计

* 向量条目携带：tenant_id / secrecy_level / acl_scope / source_doc_id
* 检索前做权限过滤，检索后记录 evidence（doc_id + chunk_id + hash）

### 成本控制策略

* 先检索后生成（retrieve-then-generate）
* 输出严格 JSON Schema
* 业务节点触发：只有进入 Gate 才跑高成本任务（比如合同风险）

# 9. 关键业务规则（必须写成可配置）

## 9.1 配置表建议

* stage_definition（stage_code, name, gate_rules_json）
* score_rule（线索/机会打分规则）
* margin_policy（毛利红线按产品线/客户等级/区域）
* approval_policy（报价/合同/变更触发审批条件）
* secrecy_policy（附件密级默认与下载规则）

# 14. 状态机与关键约束（可直接落库/写代码）

## 14.1 OpportunityProject 状态机

* `stage_code`: S1~S9（来自 stage_definition）
* 规则：

  * 只能按配置的 `allowed_transitions` 推进/回退
  * 推进必须通过 `gate_rules_json` 校验
  * 每次变更写入 `project_stage_history`

**推荐字段约束**

* `project_code`：`tenant_id + project_code` 唯一
* `customer_id` 必填（线索转化后必须绑定）
* `owner_user_id` 必填

**索引**

* `(tenant_id, stage_code)`
* `(tenant_id, owner_user_id, stage_code)`
* `(tenant_id, customer_id)`

## 14.2 Quote / QuoteVersion 状态机

### quote.status

* draft → reviewing → sent → (won|lost) → obsolete
* 约束：

  * `sent` 必须关联至少一个 `quote_version` 且该版本 `approval` 通过
  * `won` 必须存在 `contract` 或 `erp_order_link`（配置决定）

### quote_version（隐式状态）

* 通过 `approval_flow.status` 体现（draft/reviewing/approved/rejected）

**唯一约束**

* `(tenant_id, quote_id, version_no)` 唯一
* `quote.current_version_no` 指向最新版本（事务内维护）

**索引**

* `(tenant_id, quote_id)`
* `(tenant_id, project_id)`
* `(tenant_id, created_at desc)`

## 14.3 Contract / ContractVersion 状态机

* contract.status: draft → reviewing → signed → terminated
* 约束：

  * `signed` 必须 `contract_version` 审批通过且存在 `doc_attachment_id`

## 14.4 ChangeRequest 状态机（强制闭环）

* draft → reviewing → approved → implemented
* 或 rejected
* 约束：

  * `approved` 后必须在一定时间内 `implemented`（可做SLA）

# 19. RAG 权限过滤与证据链（实现要点）

## 19.1 向量条目字段（必须存）

* `tenant_id`
* `source_doc_id`
* `secrecy_level`
* `acl_scope`：如 `{"type":"project","id":123}` 或 `{"type":"department","id":9}`
* `chunk_hash`
* `text`

## 19.2 检索流程（强制）

1. 根据用户身份算出 `allowed_scopes`（项目/客户/部门/共享）
2. Milvus 检索时加过滤：tenant + scope + secrecy
3. 返回 chunk 后写入 `ai_result.evidence_json`：doc_id + chunk_hash + 摘要片段（短 snippet）

> 这样你们能做到：AI 给的每条建议都能“点开看到依据”。

## A2. 核心事务与并发策略（必须这样写）

### A2.1 创建新报价版本（从 base 复制 + 应用变更）

**事务边界：必须单事务**

1. `SELECT ... FROM quote WHERE id=? FOR UPDATE` 锁住报价主单
2. 取 `new_version_no = current_version_no + 1`
3. 插入 `quote_version(new_version_no)`
4. 复制 base 的 `quote_line` 批量插入
5. 应用 `changes`：更新行/新增行/删除行（建议用 patch 语义）
6. 重新计算：`price_total / tax_total / margin`
7. 更新 `quote.current_version_no = new_version_no`

**冲突错误**

* 并发创建导致版本号冲突：抛 `40901 version_conflict`


---

### A2.2 提交审批（报价版本进入 reviewing）

**前置校验（Gate）**

* 必须有关联 `cost_snapshot_id`
* 必须毛利≥红线（按 margin_policy）
* 必须已生成 AI 风险报告（可配置是否强制）

**事务**

* 写 `approval_flow` + `approval_task` 首节点
* 更新 `quote_version.status = reviewing`

---

### A2.3 审批通过/驳回

**通过**

* 更新 `approval_flow=approved`
* `quote_version.status=approved, approved_at=now()`
* 写 outbox：`quote.approved`

**驳回**

* `quote_version.status=rejected`
* 记录 comment + diff

---

### A2.4 发送报价（必须留痕 + 事件）

**规则**

* 只能发送 `approved` 状态的版本（或你配置允许经理强制发送）
* 发送成功后：`quote_version.status=sent`，`quote.status=sent`（若还未 sent）
* 写 `quote_send_log`
* outbox：`quote.sent`（给ERP/BI/钉钉/邮件网关）

# B. 权限域开发文档（ACL / 密级 / 字段脱敏 / RAG 权限继承）

离散制造CRM的权限**不是“能不能看一条记录”**这么简单，还要：

* **客户/项目范围**
* **附件密级**
* **字段级（成本/毛利/底价）**
* **AI 检索继承权限（否则AI越权泄密）**

## B1. 权限模型：RBAC + ACL Share + Secrecy

### B1.1 组织与角色

* `department`（树）
* `user_department`（多对多）
* `role` / `user_role`
* `role_permission`（页面/接口权限点）

### B1.2 ACL 分享（对象级）

`acl_share`

| 字段                                                    | 说明                                             |
| ----------------------------------------------------- | ---------------------------------------------- |
| tenant_id                                             |                                                |
| biz_type (customer/project/quote/contract/attachment) |                                                |
| biz_id                                                |                                                |
| to_type (user/department/role)                        |                                                |
| to_id                                                 |                                                |
| rights_json                                           | `{"read":true,"write":false,"download":false}` |
| expires_at                                            |                                                |

**计算规则**

* 默认：owner 可读写
* 部门：同部门可读（可配置）
* 分享：按 acl_share 叠加权限

### B1.3 附件密级（Secrecy）

`attachment.secrecy_level`：public/internal/restricted
**默认策略** `secrecy_policy(doc_type -> default_level)`
**下载规则**

* restricted：必须具备 `download=true` 且在 project/contract 授权范围
* internal：read 即可下载（可按你们更严格）
* public：租户内可下载

## B2. 字段级脱敏（价格/成本/毛利）

建议做成**“策略引擎”**，否则后面写死会很痛。

### B2.1 脱敏策略配置（示例）

`field_mask_policy`

```json
{
  "entity": "quote_version",
  "fields": ["cost_snapshot_id","margin_rate","margin_amount"],
  "allow_roles": ["finance","quote_manager","admin"],
  "allow_if": [
    {"type":"is_project_owner"},
    {"type":"has_acl_right","right":"view_cost"}
  ],
  "mask_mode": "null"   // 或 "range"
}
```

### B2.2 返回层处理

* 在 DTO/Serializer 层，对字段做 mask
* 同时在日志中记录“mask发生”（便于排查权限）

## B3. RAG 权限继承（必须落到实现）

向量条目必须带 scope（否则没法过滤）。

### B3.1 向量元数据字段（存入Milvus）

* `tenant_id`
* `source_doc_id`
* `secrecy_level`
* `scope_type`：project/customer/department/public
* `scope_id`
* `chunk_hash`

### B3.2 检索过滤公式（概念）

用户可检索的 scope 集合：

* public（tenant 内公共）
* 用户所在 department（若策略允许）
* 用户是 owner 或参与者的 project/customer
* acl_share 授权的对象 scope

检索时过滤：

* `tenant_id == T`
* `scope in allowed_scopes`
* `secrecy_level <= user_max_level`（或按密级规则）

> 这样做，AI 永远不会检索到“你看不到的合同/报价/图纸”。

## C3. 任务编排（统一入口 + Worker 执行）

### C3.1 统一创建任务接口

`POST /api/v1/ai/tasks`

```json
{
  "task_type": "quote_risk",
  "biz_ref": { "type": "quote_version", "id": 8893 },
  "inputs": {
    "include_docs": true,
    "include_similar_projects": true,
    "include_cost_snapshot": true
  },
  "budget": { "max_tokens": 1800, "max_cost": 0.2 }
}
```

### C3.2 Worker 执行流程（最重要）

1. `ai_task` 取 queued → running（加锁，避免重复跑）
2. **构造上下文**：

   * 业务数据：quote_version/lines/cost_snapshot
   * 文档：从 attachment_link 获取 + 文本抽取索引
   * RAG：按权限过滤检索 topK
3. 拼 prompt（模板 + context）
4. 调模型
5. 校验输出 JSON Schema
6. 写 `ai_result`（含 evidence）
7. 更新 `ai_task` token/cost/status

**失败策略**

* Schema 校验失败：可允许“自修复重试一次”（用更严格提示让其输出合规JSON）
* 超预算：直接 fail（返回预算不足错误码）

# D. “回填机制”（AI建议如何变成业务数据）

你们要的不是“看一眼报告”，而是**一键转为可编辑草案**。

## D1. 回填策略

* AI 结果不直接写 quote_line
* 生成 `draft_patch`（JSON Patch 或你自定义 patch）
* 用户在前端确认后应用（进入“新版本生成”流程）

## D2. 回填接口

`POST /api/v1/quote_versions/{vid}/apply_ai_patch`

```json
{ "ai_task_id": 50012, "mode": "create_new_version" }
```

后端动作：

* 读取 ai_result.recommended_quote_changes
* 走 A2.1 的“创建新版本事务”（base=当前版本）
* 应用 patch → 生成新版本

# E. 你们团队“照着写”就不会翻车的三条硬约束

1. **版本化必须用事务锁主单**（否则并发就炸）
2. **AI 必须结构化输出 + Schema 校验 + 证据链**（否则无法验收/无法落地）
3. **RAG 必须权限过滤**（否则越权泄密是高危事故）
